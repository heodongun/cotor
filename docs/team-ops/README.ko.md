# 팀 운영 & 온보딩 패키지

이 패키지는 팀 운영 일관성을 높이고, 신규 기여자의 온보딩을 빠르게 진행하며, 역할별 전달 산출물을 재사용할 수 있게 하기 위한 문서 모음입니다.

## 포함 문서

- `onboarding-checklist.md`: Day-0~Day-14 온보딩 체크리스트
- `operating-cadence-template.md`: 주간/격주 운영 리듬 템플릿
- `handoff-template.md`: 장애 대응, 릴리스 담당 전환, 휴가 인수인계를 위한 재사용 템플릿
- `FEATURE_OWNER_TEMPLATE.md`: 기능 담당자 체크리스트와 인수인계 템플릿
- `REVIEWER_TEMPLATE.md`: 리뷰 절차, 블로킹 이슈, 승인 형식
- `MAINTAINER_TEMPLATE.md`: 주간/릴리즈 운영 리듬과 저장소 관리 체크리스트
- `README.md`: 영어 원본 안내 문서

## 대상 역할

- 티켓을 처음부터 끝까지 맡아 전달하는 기능 담당자
- 코드/문서/QA 관점으로 검토를 수행하는 리뷰어
- 문서, 릴리즈 흐름, 운영 리듬을 유지하는 메인터이너

## 시작 순서

1. `../README.md`, `../QUICK_START.md`, `../FEATURES.md`, `../DESKTOP_APP.md`를 먼저 읽습니다.
2. `./gradlew test` 또는 현재 범위에 맞는 검증 명령으로 로컬 빌드를 한 번 확인합니다.
3. 기본 전달 루프를 따릅니다: 재현 -> 계획 -> 수정 -> 검증 -> 리뷰 -> 머지.
4. 온보딩 체크리스트를 완료한 뒤 현재 역할에 맞는 템플릿을 고릅니다.

## 권장 사용 방법

1. 신규 기여자마다 **온보딩 버디**를 지정합니다.
2. 템플릿을 이슈/PR/위키 도구에 복사한 뒤 프로젝트별 항목만 수정합니다.
3. 매월 회고를 통해 실제 운영 방식과 문서가 어긋난 부분을 업데이트합니다.

## 관리 책임

- 1차 오너: 저장소 메인테이너
- 점검 주기: 월 1회(또는 주요 프로세스 변경 직후)
- 업데이트 트리거 예시:
  - 릴리스 주기 변경
  - 온콜/트리아지 로테이션 변경
  - CI/CD 또는 브랜치 정책 변경

- [ ] 1주 이내 문서/툴링 소규모 PR 1건 제안
- [ ] 최소 1회 릴리스/트리아지 사이클 섀도잉
- [ ] cadence/handoff 템플릿으로 1회 운영 사이클 리드

## 공통 운영 리듬

- 매일: 범위, 검증, 블로커 상태를 포함한 짧은 비동기 상태 업데이트를 남깁니다.
- 티켓마다: 수정 전 재현 근거를 기록하고 수정 후 검증 근거를 남깁니다.
- 리뷰마다: 블로킹 이슈를 먼저 적고, 그 다음 잔여 리스크와 검증 공백을 적습니다.
- 릴리즈/문서 갱신마다: 상위 문서 진입점을 함께 갱신해 신규 기여자가 흐름을 바로 찾을 수 있게 합니다.

## 공통 복사 템플릿

### 비동기 상태 업데이트

```md
범위: <티켓 또는 문서 범위>
변경: <오늘 바뀐 내용>
검증: <명령 또는 확인 절차>
리스크/블로커: <없음 또는 짧은 메모>
```

### 리스크 메모

```md
리스크: <구체적인 실패 가능성>
영향: <누가/무엇이 영향받는지>
완화: <검증, 롤백, 범위 축소 방안>
담당: <역할>
```
